# 概要

- このプロジェクトはテキスト, URL, 画像を複数デバイス間でシェアするためのwebアプリを開発するものである
- 本アプリは僕が個人的に利用するのみのものであり、将来にわたってサービス展開することはない
- アプリ名は「sharebox」である

# ざっくり機能

## 認証

- なんかいい感じに

## テキストの共有

- クリップボードアイコンをクリックでテキストをコピー可能

## ファイル共有

- 画像, 動画はプレビューを表示する
- そのほかのファイル形式については、それぞれのファイル形式に合わせたアイコンとファイル名, ファイルサイズを表示する
- 「ダウンロード」のテキストリンクをクリックすることで対象ファイルをダウンロード可能

## URL共有

- URLのOGPを表示する
- URLまたはOGPカードをクリックすることでそのページへ遷移（別タブを開く）
- クリップボードアイコンをクリックでURLをコピー可能

# sharebox プロジェクト原則

  ## 1. スコープと利用前提
  - 本アプリは開発者本人のみが利用する個人ツールであり、マルチユーザー化・サービス展開は将来にわたって行わない。
  - 認証は Google OAuth によるシングルユーザー(特定メールのホワイトリスト)を前提とし、全ページを認証必須とする。公開共有機能は設けない。
  - 機能範囲は memo.md に定義された「テキスト共有 / ファイル共有 / URL(OGP)共有」の3つに限定し、スコープクリープを禁ずる。

  ## 2. コスト制約(ゼロコスト運用)
  - 本番運用の月額コストは常に 0 円でなければならない。無料枠を超過する設計は採用しない。
  - ドメインは無料サブドメイン(例: *.pages.dev)を使用する。独自ドメインは取得しない。
  - 採用可能なインフラは Cloudflare 無料枠(Pages / Workers / D1 / R2 / KV)および既に契約済みの無料枠サービスに限る。自宅サーバーや有料サービスは使用しない。
  - ファイル1つあたりの上限は 50MB、総ストレージ使用量は概ね 1GB を想定上限とする。

  ## 3. 技術スタック(固定)
  - ホスティング: Cloudflare Pages
  - フレームワーク: SvelteKit(Cloudflare adapter)
  - 言語: TypeScript(strict モード必須)
  - DB(メタデータ): Cloudflare D1
  - ファイルストレージ: Cloudflare R2(エグレス無料であることが本構成の要)
  - 認証: Arctic + Lucia による Google OAuth 自前実装、ホワイトリスト1ユーザー
  - スタイリング: Tailwind CSS v4、DESIGN.md のトークンを CSS 変数として定義
  - フォント: Inter を self-host(DESIGN.md 指定のフォールバック順に準拠)
  - OGP 取得: Cloudflare Workers で HTML を fetch・パース、結果を D1 にキャッシュ(外部 API 依存禁止)
  - 上記以外のスタックを採用する場合は、本原則の改訂を伴うこと。

  ## 4. パフォーマンス原則
  - Core Web Vitals の "Good" 閾値(LCP < 2.5s, INP < 200ms, CLS < 0.1)を常に満たすこと。
  - 初期JSバンドルは可能な限り小さく保ち、不要な依存追加を避ける。クライアント側状態管理ライブラリは原則導入しない(SvelteKit 標準で足りる範囲にとどめる)。
  - 画像は適切にフォーマット変換・遅延読み込みを行う。動画プレビューも初期ロードをブロックしない。
  - すべての動的処理はエッジ(Cloudflare)で完結させ、オリジンサーバーへの往復を作らない。

  ## 5. デザインシステム
  - UI 実装は DESIGN.md を単一の真実源とする。色・余白・タイポグラフィ・角丸・影は DESIGN.md のトークンから逸脱しない。
  - 新しいデザイントークンやコンポーネント原則が必要になった場合、実装前に DESIGN.md を更新すること。

  ## 6. 品質とテスト
  - 主要ユーザーフロー(ログイン / テキスト投稿・コピー / ファイルアップロード・ダウンロード / URL投稿・OGP表示)は Playwright による E2E テストで保護する。
  - 非自明なロジック(OGP パーサ、認証ガード、ファイルバリデーション等)にはユニットテストを書く。
  - CI(GitHub Actions 等)で型チェック・Lint・テストを自動実行し、緑でないコードはマージ禁止。
  - 厳格な TDD は求めないが、機能追加時に「E2E と必要なユニットを同じ PR で追加する」ことを標準とする。

  ## 7. 実装プロセス
  - 実装は Claude Code(AI コーディングエージェント)主導で行う。人間は仕様策定・レビュー・意思決定に集中する。
  - 機能追加は必ず Spec Kit のワークフロー(/speckit.specify → /speckit.plan → /speckit.tasks → /speckit.implement)に従って進める。
  - 仕様・計画・タスクに反する実装は行わず、乖離が必要になった場合は先に仕様を更新する。

  ## 8. シンプルさの原則
  - 「個人用」であることを理由に雑な設計を許容しない一方、マルチテナント化・拡張性・将来の他ユーザー想定などの "過剰な一般化" も禁ずる。
  - 抽象化は2回以上の具体的重複が出現してから導入する。投機的抽象化を禁ずる。
  - 依存関係は最小限に保ち、追加時はその必要性をコミットメッセージまたは PR 説明で justify すること。
